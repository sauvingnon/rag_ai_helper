# RAG Voice Assistant

Голосовой ассистент справочной службы университета на базе Retrieval-Augmented Generation. Принимает звонки по SIP-телефонии, отвечает на вопросы об университете синтезированной речью. Поддерживает как облачные, так и полностью локальные LLM.

---

## Архитектура

```
Звонящий
    │ SIP / RTP
    ▼
┌─────────────────┐
│    Asterisk      │  SIP-сервер, маршрутизирует аудиопоток
└────────┬────────┘
         │ AudioSocket TCP (бинарный протокол, PCM 8kHz 16-bit)
         ▼
┌─────────────────────────────────────────────────────┐
│              telephony_service (порт 9093)           │
│                                                      │
│  Silero VAD → буфер речи → WAV                       │
│  barge-in детектор (пользователь перебивает ответ)   │
│  Silero TTS → PCM → AudioSocket → Asterisk           │
└────────────────────┬────────────────────────────────┘
                     │ HTTP streaming (JSON lines)
                     ▼
┌─────────────────────────────────────────────────────┐
│                ai_service (порт 8005)                │
│                                                      │
│  faster-whisper STT                                  │
│  Agentic RAG loop (DeepSeek / Groq / Ollama)         │
│    ├─ search_knowledge_base → SBERT + Cross-Encoder  │
│    └─ ask_clarification                              │
│  ChromaDB (персистентная векторная БД)               │
└─────────────────────────────────────────────────────┘

Браузерный интерфейс (опционально):
┌──────────────────┐        ┌─────────────────────────┐
│  voice_service   │ ──────▶│      ai_service          │
│  (порт 8010)     │        │                          │
│  WebSocket + TTS │        └─────────────────────────┘
└──────────────────┘
```

## Компоненты

### ai_service — мозг системы

FastAPI-сервис, который объединяет STT, RAG-пайплайн и LLM.

**STT:** `faster-whisper` (модель `small`, CPU, int8). Принимает WAV, возвращает текст. Встроенная защита от галлюцинаций (фильтрация субтитров, шумовых артефактов).

**RAG-пайплайн — двухэтапный поиск:**

1. **SBERT** (`ai-forever/ru-en-RoSBERTa`) — быстрый семантический поиск по ChromaDB. Создаёт эмбеддинги для `name + text + keywords` каждого чанка. Возвращает TOP_K кандидатов по cosine similarity.

2. **Cross-Encoder** (`DiTy/cross-encoder-russian-msmarco`) — точный реранкинг. Берёт пары `(запрос, чанк)` и выдаёт скор релевантности. Возвращает RERANK_TOP лучших. Если лучший скор ниже RERANK_THRESHOLD — возвращает `None` (запрос не релевантен базе).

**Почему именно так:** SBERT даёт скорость (approximate nearest neighbors по всей базе за миллисекунды), Cross-Encoder даёт точность (работает медленнее, но оценивает каждую пару полностью). Комбинация даёт баланс скорости и качества.

**Agentic RAG — LLM решает сам:**

Вместо того чтобы всегда искать при любом запросе, LLM получает два инструмента:

- `search_knowledge_base(query)` — вызывает RAG-пайплайн. Можно вызвать до 2 раз за ход с разными формулировками запроса.
- `ask_clarification(question)` — задаёт уточняющий вопрос, если запрос совершенно непонятен.

LLM сам принимает решение: ответить из своих знаний, поискать один раз, поискать дважды с уточнением, или попросить уточнить. Это даёт три паттерна в реальных звонках:
- Простой вопрос (`"когда основан университет?"`) → прямой ответ без поиска
- Конкретный вопрос (`"расскажи про кафедру механики"`) → один поиск
- Сложный вопрос (`"что нужно для поступления на IT и есть ли общежитие?"`) → два поиска

### telephony_service — голосовой ввод/вывод

Реализует протокол AudioSocket поверх TCP. Asterisk устанавливает TCP-соединение и гоняет бинарные фреймы:

```
[1 байт тип] [2 байта длина big-endian] [N байт payload]
0x01 UUID  — первое сообщение, идентификатор звонка
0x10 AUDIO — PCM-фрейм: 8kHz, 16-bit, signed LE, mono
0x00 HANGUP — звонок завершён
```

**VAD:** Silero VAD с чанками по 256 сэмплов (строго, иначе модель падает при 8kHz). Буферизует речь, детектирует конец реплики по паузе.

**TTS:** Silero v3 (`xenia`), 24kHz → ресэмпл до 8kHz линейной интерполяцией. Форматирование перед синтезом: телефоны разбиваются на группы с паузами (`8(3412)77-62-62` → `8, 3412, 77 62 62`), email читается вслух (`user@mail.ru` → `user, собака, mail точка ru`). Скорость речи занижена на ~12% через растяжение аудио.

**Стриминг ответа:** `telephony_service` стримит предложения из `ai_service` по одному — каждое предложение синтезируется и воспроизводится сразу, не дожидаясь полного ответа LLM. Это сокращает субъективное время ожидания.

**Barge-in:** Пока ассистент говорит, VAD слушает входящий поток. Если пользователь начинает говорить (порог 0.75) — воспроизведение прерывается, фраза собирается и немедленно обрабатывается. Эхо-окно 350мс защищает от самопрерывания.

**Персонализация:** При первом звонке ассистент спрашивает имя и обращается по нему на протяжении всего разговора. История разговора передаётся в LLM (последние 10 пар реплик).

### asterisk

Docker-образ `mlan/asterisk`. Конфигурация PJSIP — принимает SIP-регистрацию от MicroSIP или любого другого SIP-клиента. Входящие звонки маршрутизируются в `telephony_service` через `AudioSocket`.

### voice_service

Альтернативный интерфейс — браузер через WebSocket. Тот же `ai_service`, но без телефонии. Полезен для разработки и демо без SIP-клиента.

### admin_service + Garage

`admin_service` — FastAPI-сервис для управления базой знаний. `Garage` — S3-совместимое объектное хранилище (self-hosted). Позволяет хранить файлы базы знаний отдельно от контейнеров.

## База знаний

### Формат YAML

```yaml
- type: Подразделение          # категория (не участвует в эмбеддинге)
  name: Кафедра механики       # участвует в эмбеддинге
  keywords: механика, сопромат, машиностроение, кафедра  # участвуют
  text: |                      # участвует
    Кафедра создана в 2018 году путём слияния...
    Ведёт подготовку по направлениям...
  meta:
    notes: |                   # не в эмбеддинге, но попадает в контекст LLM
      Руководитель: Иванов И.И., заведующий кафедрой
      Телефон: 8 (3412) 77-60-55 доб. 3245
      Почта: mehanika@istu.ru
      Адрес: корпус 3, каб. 808
```

**Принцип:** один чанк — одна смысловая единица, один тип вопроса. Если чанк содержит 10 сотрудников с телефонами, SBERT сделает размытый вектор и поиск не сработает.

### Подготовка данных

Скрипт `db_loader/_clean_departments.py` — пайплайн очистки сырого скрапа через LLM:
- Читает YAML с сырыми данными сайта (имена всех сотрудников, телефоны, навигационные метки)
- Отправляет каждый чанк в LLM с промптом извлечения значимого контента
- Генерирует чистые `text`, `keywords`, `meta.notes`
- Совместим с любым OpenAI-совместимым API (DeepSeek, Groq, OpenAI)

`db_loader/db_loader.py` — загрузка YAML в ChromaDB:
- Рекурсивно читает все `*.yaml` из `data/`
- Строит обогащённый текст: `name + text + keywords + notes`
- Батчевый encode через SBERT (batch_size=32)
- Вставляет документы, эмбеддинги и метаданные в Chroma

**Важно:** ChromaDB привязана к конкретной SBERT-модели. При смене модели — пересоздавать базу заново.

## Варианты LLM

Сервис использует OpenAI-совместимый API. Переключение через `.env`:

### Облако — DeepSeek (дефолт)
```env
LLM_BASE_URL=https://api.deepseek.com/v1
LLM_MODEL=deepseek-chat
LLM_API_KEY=sk-...
```
Хорошее качество, низкая стоимость, стабильный tool use.

### Облако — Groq
```env
LLM_BASE_URL=https://api.groq.com/openai/v1
LLM_MODEL=llama-3.3-70b-versatile
LLM_API_KEY=gsk_...
```
Очень быстрый инференс, есть бесплатный тариф — хорош для разработки.

### Локально — Ollama
```env
LLM_BASE_URL=http://host.docker.internal:11434/v1
LLM_MODEL=qwen2.5:7b
LLM_API_KEY=ollama
```
Полностью локальный инференс. Подходит для конфиденциальных данных или offline-режима. Качество зависит от модели и железа. Tool use работает только с моделями, поддерживающими function calling.

## Конфигурация (.env)

```env
# LLM
LLM_BASE_URL=https://api.deepseek.com/v1
LLM_MODEL=deepseek-chat
LLM_API_KEY=sk-...

# Модели (HuggingFace ID для скачивания)
SBERT_MODEL=ai-forever/ru-en-RoSBERTa
CROSS_ENCODER_MODEL=DiTy/cross-encoder-russian-msmarco

# Локальные пути в контейнере (не трогать)
SBERT_MODEL_PATH=/app/services/embeddings_service/models/sbert_model
CROSS_ENCODER_MODEL_PATH=/app/services/embeddings_service/models/cross_encoder_model

# RAG параметры
TOP_K=20              # кандидатов из ChromaDB (SBERT)
RERANK_TOP=5          # финальных чанков в контекст LLM (Cross-Encoder)
RERANK_THRESHOLD=-2.0 # порог скора: ниже — считаем нерелевантным

# STT
WHISPER_MODEL=small   # tiny / base / small / medium
```

**Подбор TOP_K и RERANK_TOP:**
- `TOP_K` — чем больше база, тем выше нужен. При 260 чанках оптимально 15–25.
- `RERANK_TOP` — сколько чанков получит LLM. Больше = полнее ответ на широкие вопросы, но дороже токены. Оптимум 4–6.
- `RERANK_THRESHOLD` — `-2.0` пропускает почти всё. Повышай (до `-1.0`, `0.0`) если ассистент отвечает на нерелевантные вопросы из базы.

## Запуск

### 1. Скачать модели (один раз)

```bash
python helper_services/ai_service/dowloand.py
```

Читает `SBERT_MODEL` и `CROSS_ENCODER_MODEL` из `.env`, сохраняет в `ai_service/models/`.

### 2. Подготовить базу знаний

```bash
# Положи YAML-файлы в helper_services/db_loader/data/
# Запусти загрузчик локально (не в Docker):
cd helper_services/db_loader
python db_loader.py
# ChromaDB создастся в helper_services/db_loader/chroma_db/
# Перемести в ai_service/chroma_db/
```

### 3. Поднять сервисы

```bash
cd helper_services
docker compose up -d
```

Порядок запуска по healthcheck: `garage` → `admin_service`, `ai_service` → `telephony_service` → `asterisk`.

### 4. Подключить SIP-клиент

Настрой MicroSIP (или любой SIP-клиент) на регистрацию на `localhost:5060`. Конфиг учётных записей в `helper_services/config/pjsip.conf`.

## Отладка

| Симптом | Причина | Решение |
|---|---|---|
| Chroma ничего не находит | Пустые эмбеддинги или не та модель | Проверь, что `db_loader.py` запускался с той же SBERT что в `.env` |
| LLM отвечает мимо | Шумные чанки, размытый вектор | Один чанк = одна тема. Перепроверь `text` в YAML |
| Rerank возвращает None | Порог слишком высокий | Понизи `RERANK_THRESHOLD` до `-2.0` |
| Нет звука из Asterisk | NAT / AudioSocket не достучался | Проверь `pjsip.conf`, убедись что порт 9093 доступен |
| Silero падает на VAD | Неправильный размер чанка | VAD при 8kHz требует ровно 256 сэмплов (512 байт) |
| TTS обрезает начало | Нет lead-in тишины | В `send_audio` должен быть 60мс prepend |
| Модель не влезает в RAM | Слишком тяжёлый SBERT | Используй `small` вариант или ограничь `mem_limit` в compose |

## Потенциал и направления развития

### Качество поиска
- **Гибридный поиск (BM25 + SBERT)** — комбинировать лексический и семантический поиск. Хорошо работает для точных совпадений (коды специальностей `09.03.01`, имена) + семантики. ChromaDB поддерживает через `where` фильтры.
- **Смена SBERT** на `intfloat/multilingual-e5-large` — по бенчмаркам на русском иногда превосходит `ru-en-RoSBERTa`, особенно на длинных текстах.
- **Query expansion** — перед поиском LLM переформулирует запрос несколькими способами, поиск идёт по нескольким вариантам, результаты мержатся.

### Знания
- **Инкрементальное обновление** — сейчас база пересоздаётся целиком. Можно добавить `upsert` по ID и обновлять только изменившиеся чанки.
- **Автоматический скрапинг + очистка** — пайплайн уже есть (`_clean_departments.py`), можно запускать по расписанию при обновлении сайта.
- **Структурированные данные** — расписание, стоимость обучения, проходные баллы — лучше хранить отдельно и запрашивать через отдельный инструмент агента.

### Голос и телефония
- **GPU для TTS/STT** — Silero и Whisper на GPU дают 5–10x ускорение, задержка ответа падает с ~3с до <1с.
- **SIP-транк** — Phase 2: подключить реального оператора (Zadarma, МТТ и др.), ассистент будет отвечать на реальные звонки.
- **Лучший TTS** — `XTTS-v2` или `CoquiTTS` дают более естественную речь, но требуют GPU.
- **Несколько голосов** — разные ассистенты для разных направлений (приёмная комиссия, общежитие, библиотека).

### Архитектура
- **Локальный LLM с GPU** — `Qwen2.5-7B` или `Gemma-2-9B` на RTX 3080+ закрывает полностью offline-сценарий с приемлемым качеством.
- **Логирование звонков** — транскрипты в S3 (Garage уже есть), аналитика вопросов помогает понять что в базе не хватает.
- **Веб-интерфейс администратора** — редактирование YAML прямо в браузере, просмотр чанков, тест-запросы к RAG.
- **Многоязычность** — `ru-en-RoSBERTa` уже двуязычная, достаточно добавить английские `keywords` в чанки.

## Выводы

Проект демонстрирует рабочую архитектуру голосового RAG-ассистента для справочной службы:

- **SBERT + Cross-Encoder** — проверенная связка speed/precision для русскоязычного поиска. Заменяемо: достаточно сохранить модели в нужные папки и пересоздать базу.
- **Agentic RAG** вместо наивного "всегда искать" — LLM сам определяет нужен ли поиск и сколько раз. Это снижает латентность на простых вопросах и повышает качество на сложных.
- **Качество базы знаний важнее модели** — 40 хорошо составленных чанков работали лучше чем 2000 сырых. LLM-очистка скрапа — рабочий подход к масштабированию.
- **Гибкость LLM** — единственное изменение в `.env` переключает между облачными и локальными моделями.
- **Телефония без облака** — Asterisk + AudioSocket + SIP даёт полноценную телефонию локально, без зависимости от провайдеров коммуникаций.
