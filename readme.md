# Составляющие проекта:

- SBERT — sentence-transformer для эмбеддингов (пример: *ai-forever/ru-en-RoSBERTa*)
- Cross-Encoder — модель для rerank (пример: *DiTy/cross-encoder-russian-msmarco*)
- ChromaDB — векторная база (persisted)
- LLM — Gemini (любой LLM, который ты хочешь юзать через API)
- Whisper — модель для speech-to-text (локальная/удалённая)

# Описание pipeline:

1) У нас микросервисная архитектура - кидай запрос откуда угодно. (в нашем примере — Telegram-бот)
2) Если запрос — аудио → Whisper делает speech-to-text → получаем текст.
3) Текст (и текст из аудио) идёт в Chroma:
- - Сначала — SBERT создаёт эмбеддинги и делает nearest neighbors → получаем TOP_K.
- - Потом — Cross-Encoder берёт эти TOP_K и делает точный rerank → получаем RERANK_TOP (N) (обычно 3).
4) Эти N топ-чанков вместе с промптом пользователя подаются в LLM (Gemini) — LLM сопоставляет, формирует ответ, возвращаем пользователю.

## Почему так? 

- SBERT — быстрый и дешёвый поиск по смыслу (approximate nearest neighbors).
- Cross-Encoder — медленнее, но точнее для тонкой сортировки конкретно под запрос.
- Комбинация даёт баланс скорости и качества.

# Почему важны именно SBERT + Cross-Encoder?

- SBERT / sentence-transformer — делает стабильные, семантические векторы для больших коллекций. Если брать любую модель, но не из SBERT-подхода, результаты поиска на уровне nearest-neighbors могут быть менее релевантны.
- Cross-Encoder — принимает пару (запрос, документ) и выдаёт скор прямо для пар, что лучше для ранжирования. SBERT даёт кандидатов, Cross-Encoder их перепроверяет.
- Итог: SBERT = speed, Cross-Encoder = precision. Менять на что-то другое можно, но должно быть того же класса.

# Файлы и расположение (важно):

- *rag_ai_helper/helper_services/ai_service/models/* — локальные модели (SBERT, Cross-Encoder)

# Формат yaml 

Каждый элемент — документ. Массив объектов в файле — ок.

```yaml
- type: "product"            # не участвует в эмбеддинге, только для фильтра/человека
  name: "iPhone 15 Pro"      # участвует в эмбеддинге
  keywords: "смартфон, Apple, titanium"  # участвует
  text: |
    Новый флагман с процессором A17 Pro и титановым корпусом.
    Дисплей 120 Гц, улучшенные камеры.
  meta:
    notes: |
      Релиз: сентябрь 2023
      Цвета: Black Titanium, White Titanium

```

- *name, keywords, text* участвуют в тексте, из которого строим эмбеддинг.
- *type, meta.notes* не идут в эмбеддинг, но сохраняются в метаданных Chroma и попадут в LLM (если нужно включать в prompt).

# Шаги запуска (что делать, порядок)

1. Подготовь *.env* (см. выше). Все переменные обязательны.
2. Скачивание моделей (1 раз):
```bash
python rag_ai_helper/helper_services/download.py
```
- скрипт читает *.env* и сохраняет модели в *.../models/*.
(Если у тебя уже есть свои скрипты — ок, главное, чтобы модели в models/ лежали локально.)
3. Генерация ChromaDB (внимание: DB привязана к типу SBERT):
- Помести YAML’ы в *rag_ai_helper/helper_services/data/*.
- Запусти загрузчик:
```bash
python rag_ai_helper/helper_services/db_loader/db_loader.py
```
- В результате появится *chroma_db/*. Перемести её в *ai_service/chroma_db*. При формировании образа Docker-compose подхватит ее через bind mount.
⚠️ Важное: База векторов сформирована именно под выбранный SBERT-тип. Менять модель SBERT на другой тип нельзя — векторы несовместимы → всё сломается. Если хочешь другую модель → пересоздай DB заново с новой моделью.
4. Поднять сервисы / бот
- Запусти ai_service (FastAPI или твой сервис), который:

- - грузит SBERT и Cross-Encoder из локальных путей,

- - подключает ChromaDB,

- - предоставляет endpoint, который принимает текст/аудио и возвращает LLM-ответ.

- Запусти Telegram-bot, который пересылает запросы в ai_service.

! Аудио гоняем в base64 строке. В схемах это видно.

# Пояснения по конфигам TOP_K и RERANK_TOP

- *TOP_K* (первичный поиск SBERT): обычно 10–50. Больше — дольше и шумнее.
- *RERANK_TOP* (Cross-Encoder): 1–5. Не делай слишком большим, LLM может запороться на слишком большом контексте + расходы растут. Нормально — 3.

# Деплой и bind-mount

- *models/* и *chroma_db/* лучше монтить в контейнер как volume/bind. Это уже прописано в docker-compose файле.
- Почему: скачивать модели при каждом билде контейнера — долго.
- При смене моделей — пересобирать образ/перекладывать папки вручную. Если после того как всё устаканилось — можно автоматизировать.

# Отладка и частые проблемы

- "Модель не влезла в память" — ставь меньшую SBERT или делай батчи при encode.
- "Chroma не находит ничего" — проверь, что embeddings действительно были записаны и что векторы не пустые.
- "Rerank выдал пусто/не то" — проверь, что Cross-Encoder совместим с языком/доменом.
- "LLM отвечает фигню" — давай ему контекст: включай name/text/keywords, и чётко формируй system prompt (в prompt укажи: «используй только эти фрагменты и ответь коротко»).

# Checklist перед защитой/демо

 - *.env* заполнен и валиден (нет пустых переменных).
 - Модели скачаны в *models/*.
 - *chroma_db/* создан и лежит в нужном месте.
 - Bot настроен и может достучаться до *AI_SERVICE_URL*.
 - *TOP_K* и *RERANK_TOP* выставлены адекватно (например 15 и 3).
 - Примеры YAML с тестовыми данными в папке *data/* (и один ручной тест-запрос).
