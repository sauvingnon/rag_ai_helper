# Voice Assistant — архитектурный разбор

## Что верно в исходном видении

- **Оставить RAG-ядро** — абсолютно правильно, это переиспользуемая и рабочая часть
- **Локальный контур для real-time** — верно, дёргать Groq API в реальном времени → непредсказуемая задержка 200ms–2s только на сеть
- **WebSocket для потока голоса** — правильный транспорт

---

## Где стоит скорректировать мышление

### 1. "GPT 2-3 версии" — устаревшая формулировка

GPT-2/GPT-3 — это старые модели OpenAI, локально не запустишь. Современный открытый эквивалент:

| Модель | Размер (q4) | Качество RU | RAM |
|--------|------------|-------------|-----|
| `llama3.2:3b` | ~2 ГБ | хорошее | 6 ГБ |
| `qwen2.5:3b` | ~2 ГБ | **отличное** (специально обучена на RU) | 6 ГБ |
| `qwen2.5:7b` | ~5 ГБ | очень хорошее | 12 ГБ |
| `gemma2:2b` | ~1.5 ГБ | среднее | 4 ГБ |

Рекомендую смотреть в сторону **Qwen 2.5 3B/7B** — лучший русский язык в своём классе.

### 2. Наивный пайплайн STT → RAG → LLM → TTS имеет проблему latency

Если складывать шаги последовательно:
```
STT:    ~800ms  (faster-whisper small)
RAG:    ~200ms  (ChromaDB + SBERT)
LLM:    ~2-5s   (3B на CPU, полный ответ)
TTS:    ~500ms  (Silero)
─────────────────
Итого:  ~3.5-6.5 секунд молчания перед ответом
```

Это **неприемлемо** для голосового ассистента. Решение — **стриминг**:

```
LLM генерирует токены по одному
      │
      ▼
TTS обрабатывает первое предложение (5-15 слов)
      │
      ▼
Первый аудио-чанк уходит пользователю (~1.5-2s после STT)
пока LLM ещё генерирует следующие предложения
```

Это снижает воспринимаемую задержку до ~1.5–2 секунд — уже приемлемо.

### 3. Нужен VAD (Voice Activity Detection)

Без него ты не знаешь, когда пользователь **перестал говорить** и нужно запускать STT. Это отдельный маленький компонент, который работает постоянно.

## Предлагаемая архитектура

```
SIP звонок
    │
    ▼
Asterisk (ARI)
    │ аудио поток (RTP → PCM 8kHz/16kHz)
    ▼
Python Voice Service (WebSocket)
    │
    ├─ Silero VAD ──► детектировать конец речи
    │
    ├─ faster-whisper (small) ──► текст запроса
    │
    ├─ RAG Core (из текущего проекта, без изменений)
    │     └─ SBERT → Chroma → Cross-Encoder → топ-3 чанка
    │
    ├─ Ollama (qwen2.5:3b) ──► стриминг токенов
    │         │
    │         ▼
    ├─ Silero TTS ──► аудио-чанки по предложениям
    │
    └─ Asterisk (ARI) ──► отдать аудио обратно
```

### Что использовать для каждого слоя

| Задача | Инструмент | Почему |
|--------|-----------|--------|
| VAD | `silero-vad` | быстрый, CPU, хорошо работает |
| STT | `faster-whisper` (small/medium) | CTranslate2, в 4x быстрее обычного Whisper |
| LLM serving | `Ollama` | простой запуск, стриминг из коробки, GGUF quantization |
| TTS | `Silero TTS` | лучший русский TTS на CPU, <100ms на предложение |
| Telephony | `Asterisk ARI` | sip.conf уже есть в проекте |
| RAG | текущий код | ничего не менять |

## По ресурсам

**Минимум для PoC (CPU-only, qwen2.5:3b):**
- 16 ГБ RAM, 8 CPU cores
- ~10–15 токенов/сек → первый аудио-чанк за ~2 сек
- ~2000–3000 руб/мес (Timeweb, Selectel)

**Комфортный вариант (CPU, qwen2.5:7b):**
- 32 ГБ RAM, 16 cores
- ~8–12 токенов/сек
- ~4000–6000 руб/мес

**Быстрый вариант (GPU, T4 16GB):**
- ~80–100 токенов/сек, latency в разы ниже
- vast.ai: ~$0.20–0.35/ч (для апробации это ~2000–5000 руб/мес при умеренной нагрузке)

**Рекомендация для апробации:** стартовать на CPU 32GB (Selectel/Timeweb) с qwen2.5:3b — это даст понять реальную задержку. Если не устроит — переходить на GPU T4.

## Итого

Исходный подход в целом **правильный**, основные поправки:
1. Не "GPT 2-3", а современные open-source модели (Qwen/Llama)
2. Обязательный стриминг LLM→TTS, иначе задержка неприемлема
3. Нужен VAD как отдельный компонент
4. RAG-ядро можно взять почти без изменений
